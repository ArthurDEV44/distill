# `PostToolUse` hooks for Distill

Claude Code runs registered hooks after every tool call, including MCP tool calls. This page shows how to wire them against Distill's three tools (`auto_optimize`, `smart_file_read`, `code_execute`) so you can build your own observability or rewrite the output the model sees.

`PostToolUse` hooks live in `~/.claude/settings.json` (user-level) or `.claude/settings.json` (project-level) under a `hooks.PostToolUse` key. Each entry pairs a `matcher` (tool-name filter) with a list of hooks to execute.

## Basic snippet

Copy the following into `~/.claude/settings.json` to fire a shell script after every Distill tool call:

```jsonc
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "mcp__distill-mcp__*",
        "hooks": [
          {
            "type": "command",
            "command": "/home/<you>/bin/distill-audit.sh"
          }
        ]
      }
    ]
  }
}
```

**Manual verification.** Paste the snippet into your `settings.json`, restart Claude Code, and trigger `auto_optimize` from a session. Your script receives the hook payload on stdin. If `distill-mcp` isn't connected, the matcher is inert — nothing is ever spawned.

### Exact matcher semantics

Claude Code interprets `matcher` through the following pipeline (`claude-code/utils/hooks.ts:1346-1383`):

1. Bare `*` → "match everything".
2. Simple string or pipe-separated list (letters, digits, `_`) → exact match (`Write` or `Write|Edit`).
3. Anything else → treated as a regex applied via `regex.test(toolName)` — a substring match, not anchored.

The `mcp__distill-mcp__*` form above works by side-effect: `*` in regex means "zero or more of the previous char" (here `_`), and `test()` matches anywhere in the string, so a tool name like `mcp__distill-mcp__auto_optimize` passes. For a more explicit, less-fragile filter, anchor the regex:

```jsonc
{ "matcher": "^mcp__distill-mcp__" }
```

Or list the three tools explicitly:

```jsonc
{
  "matcher": "mcp__distill-mcp__auto_optimize|mcp__distill-mcp__smart_file_read|mcp__distill-mcp__code_execute"
}
```

## Hook output contract

Your script receives the `PostToolUse` JSON payload on stdin (standard fields: `session_id`, `tool_name`, `tool_input`, `tool_response`, `cwd`, etc.). It can either print nothing to stdout — the tool result passes through unchanged — or emit a JSON object that follows the Claude Code hook output schema documented at `claude-code/utils/hooks.ts:425-440`:

```jsonc
{
  "hookSpecificOutput": {
    "hookEventName": "PostToolUse",
    "additionalContext": "optional string injected into the next turn",
    "updatedMCPToolOutput": "anything you want"
  }
}
```

**The `updatedMCPToolOutput` channel** (`claude-code/utils/hooks.ts:350, :370, :645-649`) is MCP-specific: when present, Claude Code replaces the tool output with this value before handing it to the model. That is the hook point for, say, re-compressing an output, wrapping it in a `[DISTILL:COMPRESSED]` marker, or filtering it before it enters the context.

## Three concrete use cases

### 1. Audit / logging

The script writes every Distill tool call into a local file — handy for counting how often `auto_optimize` fires during a workday.

```sh
#!/bin/sh
# ~/bin/distill-audit.sh
# stdin: PostToolUse hook JSON. Log and let it flow through.
payload=$(cat)
tool=$(printf '%s' "$payload" | jq -r .tool_name)
ts=$(date -u +%Y-%m-%dT%H:%M:%SZ)
printf '%s\t%s\n' "$ts" "$tool" >> "$HOME/.claude/distill-audit.log"
# No JSON on stdout → the tool output is not modified.
```

### 2. Re-compressing a large output

When a `code_execute` output exceeds your threshold, re-compress it with an external tool (or a second Distill call) and return it via `updatedMCPToolOutput` so the model only sees the short version.

```sh
#!/bin/sh
# ~/bin/distill-recompress.sh
payload=$(cat)
raw_output=$(printf '%s' "$payload" | jq -r '.tool_response.content[0].text // empty')
size=${#raw_output}
if [ "$size" -lt 2000 ]; then
  # Too short to benefit from re-compression — let it through.
  exit 0
fi

compressed=$(printf '%s' "$raw_output" | your-compressor-cli --max-lines 40)
# Emit valid JSON per the hookSpecificOutput contract.
printf '{"hookSpecificOutput":{"hookEventName":"PostToolUse","updatedMCPToolOutput":%s}}' \
  "$(printf '%s' "$compressed" | jq -Rs .)"
```

### 3. Local telemetry

Emit Prometheus / JSONL metrics on every compressed tool call, typically to measure token savings over time:

```sh
#!/bin/sh
# ~/bin/distill-metrics.sh
payload=$(cat)
tool=$(printf '%s' "$payload" | jq -r .tool_name)
input_size=$(printf '%s' "$payload" | jq -r '.tool_input | tostring | length')
output_size=$(printf '%s' "$payload" | jq -r '.tool_response.content[0].text // "" | length')
ratio=$(awk "BEGIN { if ($input_size > 0) print $output_size / $input_size; else print 0 }")
printf '{"ts":"%s","tool":"%s","in":%s,"out":%s,"ratio":%s}\n' \
  "$(date -u +%Y-%m-%dT%H:%M:%SZ)" "$tool" "$input_size" "$output_size" "$ratio" \
  >> "$HOME/.claude/distill-metrics.jsonl"
```

## Graceful degradation on invalid JSON

If your script prints something to stdout that is not valid JSON — or JSON that fails the output-schema validation — Claude Code does NOT block the tool call. The parser:

- Throws on `JSON.parse` → `claude-code/utils/hooks.ts:447-450` returns `{ plainText: stdout }`, the hook is treated as "no structured output", and the original tool result passes through to the model unchanged.
- Parses but violates the schema → `claude-code/utils/hooks.ts:391-396` returns a `validationError` rendered in debug mode, pointing at each offending field. The original tool result still passes through unchanged.

**Net result:** a broken hook degrades your telemetry but does not break the session. This is intentional — you can iterate on your script without risking your Claude Code shell.

## Operational notes

- **The matcher only fires for connected tools.** If Distill isn't registered in your `settings.json` yet (via `distill-mcp setup --claude`), the `mcp__distill-mcp__*` matcher matches nothing and the hook sleeps — no process is spawned needlessly.
- **Latency.** Each hook adds a fork + exec on the hot path of tool calls. Keep scripts lean (the `sh` examples above are intentional — prefer them over a heavy interpreter startup).
- **Security.** The stdin payload carries the tool arguments, including possibly file paths and content excerpts. Don't forward it to an external service without explicit filtering — the scripts are local for a reason.
- **Relationship to Distill's `PreCompact` hook.** If you already ran `distill-mcp setup --install-precompact-hook`, that one lives under `hooks.PreCompact` — not under `hooks.PostToolUse`. The two coexist without interference.
